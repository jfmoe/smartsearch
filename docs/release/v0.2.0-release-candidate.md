# @jfmoe/smart-search 0.2.0 release candidate handoff

## Candidate identity and evidence boundary

- Release identity: `@jfmoe/smart-search@0.2.0`.
- Read-only upstream baseline:
  `667c465d0f6ea16a423f03c434f94e21505d3595`.
- This file is a release handoff and checklist, not a publication record. It
  does not assert that any npm version, tag, workflow, or GitHub Release has
  already been created.
- The complete candidate commit SHA is intentionally not self-referenced here:
  a commit cannot contain its own final SHA. The full SHA is recorded only in
  this ticket's issue/PR evidence ledger after this handoff commit exists.
- Candidate scope is frozen. Any product, architecture, provider-contract,
  metadata, workflow, or Skill-contract change creates a new candidate and
  invalidates this handoff's prior gate evidence.

## Entry gates

Before an authorized release operator begins `0.2.0-beta.1`, record in the
issue/PR evidence ledger that all of the following passed for the exact
candidate SHA:

1. `npm test`, including Skill mirror, release metadata and identity policy,
   workflow policy, wrapper, and `npm pack --dry-run` package-content checks.
2. The release/handoff contract test and all targeted release tests.
3. The limited local live acceptance: `doctor`, one main-search request, one
   docs capability `resolve-library-id` health check, and one web fetch. Record
   only capability, provider, success status, result count or URL host, and
   failure class when applicable.
4. A standards/spec review of the candidate range, including scope freeze,
   SHA-evidence model, secret redaction, stop points, release order, and
   recovery boundaries.

Do not continue to any release stage if an entry gate fails, is missing, cannot
be tied to the exact candidate SHA, or its capability is unavailable. Keep the
issue open and report the real blocker; do not substitute another capability.
The `beta.2` stage does not repeat local live acceptance when the code and
provider configuration contract are unchanged since `beta.1`.

## Ordered, human-in-the-loop release checklist

Only an explicitly authorized release operator performs the following actions.
This repository's candidate preparation must not publish, dispatch a workflow,
change registry state, create a tag, or create a GitHub Release.

### 1. Traditional-auth preview: `0.2.0-beta.1`

1. Confirm the package version and the full candidate SHA, and confirm via
   read-only registry queries that `0.2.0-beta.1` is not already public.
2. From that immutable SHA, publish the traditional-auth preview as
   `0.2.0-beta.1` with dist-tag `next`; do not make `latest` point to a preview.
3. Read back the registry package name, version, dist-tag, tarball integrity,
   and provenance; capture the publish workflow or operator evidence in the
   evidence ledger.
4. Only after all read-backs agree, create an immutable `v0.2.0-beta.1` tag
   targeting the same SHA and create the corresponding GitHub prerelease.

If authentication, registry publication, read-back, provenance, tag creation,
or GitHub Release evidence fails, **Do not continue**. Stop before Trusted
Publishing/OIDC setup or `0.2.0-beta.2`.

### 2. OIDC/provenance preview: `0.2.0-beta.2`

1. After `beta.1` evidence is complete, configure the npm Trusted Publisher
   manually for this repository and its designated publish workflow, with no
   environment scope; keep the permission limited to npm publishing.
2. Manually dispatch the preview workflow using the exact same full candidate
   SHA and version `0.2.0-beta.2`. The workflow must publish only to `next`
   using OIDC/provenance.
3. Read back the registry name, version, dist-tag, integrity, provenance, and
   workflow run evidence.
4. Only after all read-backs agree, create immutable `v0.2.0-beta.2` tag and
   GitHub prerelease evidence for the same candidate SHA.

If the Trusted Publisher binding, workflow, registry result, provenance,
read-back, tag, or GitHub Release evidence fails, **Do not continue**. Stop
before creating `v0.2.0` or publishing stable.

### 3. Stable release: exact `v0.2.0`

1. Reconfirm that both previews, all entry gates, and the frozen candidate SHA
   remain valid.
2. Create exact immutable tag `v0.2.0` at that candidate SHA. The stable
   workflow validates the tag and all package metadata before it publishes
   `0.2.0` to `latest` and creates the GitHub Release from the versioned notes.
3. Read back the registry name, stable version, `latest` dist-tag, integrity,
   provenance, workflow run, immutable tag target, and GitHub Release target.
4. Verify again that `latest` must not point to a preview; it may point to
   `0.2.0` only after every stable read-back agrees.

If tag validation, workflow execution, publication, any registry read-back,
provenance, or GitHub Release target check fails, **Do not continue**. The
candidate has not earned release completion evidence.

## Evidence ledger and recovery rules

For every public version, the issue/PR evidence ledger and its GitHub Release
must identify the candidate SHA, version, tag, registry name, dist-tag,
integrity, provenance result, and GitHub Release target, plus a workflow run
where a workflow performed the publish or the designated traditional-auth
operator evidence for `beta.1`. Record only redacted status facts: never record
credentials, one-time passwords, API keys, configuration values, headers, or
command output that could reveal them.

Public npm versions must not be overwritten or republished, and release tags
must not be moved. Never delete-and-reissue a public version to repair a failed
stage. Stop the sequence, then recover with npm deprecation when appropriate,
safe dist-tag correction that still leaves `latest` off previews, or a new
version (for example `0.2.1`). A stable `0.2.0` fault is repaired by a later
patch release, never by altering `0.2.0`.
